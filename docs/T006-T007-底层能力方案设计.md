# T006/T007 底层能力方案设计

## 1. 目标与范围

本方案覆盖两项底层能力：

- T006：Prompt 优化引擎 MVP
- T007：参数真实映射设计

目标不是“把用户输入改写得更长”，而是建立一层稳定的中间语义层，让前端参数、参考图、模式选择都能真实影响最终生成质量，并且不把模型能力伪装成不存在的 UI 开关。

当前仓库现状：

- 前端基本将 `customPrompt` 直接透传
- 服务端 `utils/prompt-builder.js` 仍以字符串拼接为主
- 参数层目前只有 `sizeTemplate -> size`
- “自动”只存在于交互层，尚未映射为可执行策略

结论：后续实现应优先补上 `Intent -> PromptSpec -> ProviderRequest` 三层，而不是继续堆 prompt 文本。

## 2. Prompt 优化引擎 MVP

### 2.1 设计目标

MVP 目标：

- 保留用户原始意图，不擅自改题
- 补全对生成质量真正有帮助的视觉槽位
- 将参考图和参数纳入统一决策
- 在服务端完成安全过滤和最终组装
- 输出结构化结果，便于后续扩展到视频/Agent 模式

### 2.2 输入定义

建议统一输入为 `GenerationIntent`：

```json
{
  "rawPrompt": "高端护肤精华海报，冰川蓝，高级感",
  "mode": "auto",
  "preference": {
    "aspectRatio": "9:16",
    "clarity": "hd",
    "style": "product-editorial",
    "safeLevel": "strict"
  },
  "referenceInfo": {
    "imageUrl": "https://...",
    "orientation": "portrait",
    "dominantColors": ["ice-blue", "silver"],
    "styleHints": ["clean", "luxury"],
    "subjectHints": ["bottle"]
  }
}
```

其中：

- `rawPrompt`：用户原始文本，永远保留
- `mode`：`manual | auto | agent`
- `preference`：显式参数，优先级高于自动推断
- `referenceInfo`：MVP 可先接收前端传入的基础信息；后续再接入图像分析器

### 2.3 输出定义

建议输出 `PromptSpec`，而不是只返回一个字符串：

```json
{
  "subject": "护肤精华瓶",
  "scene": "冰川质感台面与冷雾环境",
  "style": "高端商业产品摄影，极简编辑感",
  "composition": "主体居中偏下，保留上方文案留白",
  "camera": "85mm 产品特写，轻微俯拍",
  "lighting": "冷白主光 + 柔和轮廓光",
  "material": "玻璃、金属、凝霜、水珠",
  "color": "冰川蓝、银白、低饱和冷色",
  "quality": "高细节，边缘干净，真实反射",
  "negativePrompt": "低清晰度，模糊，脏污背景，畸形瓶身，额外文字，水印",
  "finalPrompt": "护肤精华瓶，置于冰川质感台面与冷雾环境，高端商业产品摄影，极简编辑感，主体居中偏下，保留上方文案留白，85mm 产品特写，轻微俯拍，冷白主光加柔和轮廓光，玻璃与金属材质，凝霜与水珠细节，冰川蓝与银白配色，高细节，边缘干净，真实反射"
}
```

`finalPrompt` 给模型直接用；其余字段用于调试、评估、后续视频化。

### 2.4 处理步骤

#### 步骤 1：去歧义

输入清洗后，先做轻量语义归一：

- 合并重复词和口语填充词
- 识别主体、场景、风格词、用途词
- 识别冲突表达，如“极简但元素很多”“夜景但阳光直射”
- 对明显不完整输入补全最小骨架，如只有“护肤品海报”时补一个默认场景框架

MVP 建议使用规则+词典，不强依赖额外 LLM。

#### 步骤 2：主题提取

抽取主语义槽位：

- 主体：人 / 产品 / 动物 / 建筑 / 食物 / 抽象概念
- 任务类型：海报 / 封面 / 宣传图 / 情绪图 / 电商图
- 主场景：室内 / 户外 / 棚拍 / 抽象空间
- 核心卖点：高级感、治愈感、科技感、节日感等

输出一个 `IntentSummary`，作为后续补全的依据。

#### 步骤 3：风格补全

当用户未明确风格时，根据 `mode + posterType + referenceInfo + preference.style` 自动补全。

建议内置一组风格模板：

- `product-editorial`
- `cinematic-realism`
- `flat-graphic`
- `fashion-campaign`
- `warm-lifestyle`
- `tech-futuristic`

补全原则：

- 用户已写明风格时，只增强，不覆盖
- 有参考图时，以参考图风格为主，用户风格为辅
- `auto` 模式下默认避免“风格词过多”，控制在 1 到 2 个风格主轴

#### 步骤 4：构图 / 镜头 / 光线 / 材质补全

这是提升画质最关键的一层。MVP 建议按“主体类型”补默认视觉槽位：

| 主体类型 | 构图默认 | 镜头默认 | 光线默认 | 材质/细节默认 |
|---|---|---|---|---|
| 产品 | 主体突出，留文案区 | 50mm/85mm 特写 | 主光+轮廓光 | 玻璃、金属、水珠、反射 |
| 人像 | 三分构图或居中 | 35mm/50mm | 侧逆光/柔光 | 皮肤质感、发丝、服装纹理 |
| 食物 | 桌面构图或微距 | 50mm/100mm | 柔和顶光 | 蒸汽、酱汁、表面纹理 |
| 动物 | 眼神聚焦 | 85mm 长焦感 | 自然光 | 毛发层次、环境颗粒 |
| 建筑/空间 | 透视明确 | 广角 | 环境光 | 材质纹理、结构边缘 |

补全原则：

- 缺失时补，不强行覆盖用户已有描述
- 比例会反向影响构图，例如 `9:16` 默认强调纵深和上下留白
- 海报场景优先考虑“可排版性”，不能只追求纯摄影感

#### 步骤 5：安全过滤

安全层建议保留现有 `utils/sensitive-filter.js` 思路，但前移到结构化阶段：

- `strict block`：政治、暴力、成人、违法等高风险内容直接拒绝
- `soft rewrite`：边缘词替换为中性视觉表达
- 对优化后新增词也要再次过滤，避免补全过程引入风险词

输出：

- `sanitizedPromptSpec`
- `safetyResult`
- 用户可见错误文案或替换说明

### 2.5 Prompt 组装策略

MVP 不建议输出超长散文式 prompt，建议使用稳定顺序：

`主体 -> 场景 -> 风格 -> 构图 -> 镜头 -> 光线 -> 材质 -> 配色 -> 质量词`

理由：

- 更接近当前图像模型对视觉槽位的理解顺序
- 便于局部调试
- 便于后续复用到视频 prompt grammar

### 2.6 三组示例

#### 示例 1：产品海报

原始输入：

`高端护肤精华海报，冰川蓝，高级感`

优化后 prompt：

`护肤精华瓶，置于冰川质感台面与冷雾环境，高端商业产品摄影，极简编辑感，主体居中偏下，保留上方文案留白，85mm 产品特写，轻微俯拍，冷白主光加柔和轮廓光，玻璃与金属材质，凝霜与水珠细节，冰川蓝与银白配色，高细节，边缘干净，真实反射`

负向 prompt：

`低清晰度，模糊，脏污背景，瓶身变形，廉价塑料感，额外文字，水印`

#### 示例 2：生活方式场景

原始输入：

`一只橘猫在窗边睡觉`

优化后 prompt：

`一只橘猫蜷缩在木质窗台上安静睡觉，室内清晨生活方式场景，治愈系写实摄影，主体位于画面右下，窗外虚化绿景形成背景层次，50mm 近景，柔和晨光从侧后方照入，毛发蓬松细腻，木材与棉布纹理清晰，暖米色与浅绿色配色，画面安静，细节自然，无遮挡，无额外文字`

负向 prompt：

`多只猫，畸形肢体，脏乱环境，过曝，模糊，卡通化，水印，文字`

#### 示例 3：参考图驱动的品牌视觉

输入：

- 原始文本：`运动品牌春季海报，女生晨跑，干净有力量`
- 模式：`auto`
- 参考图信息：竖版、蓝白主色、玻璃长廊、真实摄影感

优化后 prompt：

`年轻女性在清晨穿过玻璃长廊慢跑，运动品牌春季 campaign 海报，真实商业摄影，干净有力量的品牌表达，竖版构图，主体位于中景，保留顶部与左侧排版留白，35mm 运动抓拍，轻微动势模糊但人物主体清晰，清晨冷白自然光穿过玻璃结构，服装面料与鞋面纹理清晰，蓝白主色，空气感通透，画面克制，高质感，无杂乱背景`

负向 prompt：

`多人拥挤，动作僵硬，背景脏乱，过度运动模糊，低清晰度，错误肢体，文字，水印`

## 3. 参数真实映射

核心原则：

- 前端只暴露能真实生效的参数
- 参数分为“模型原生参数”和“复合策略参数”
- 参数冲突由服务端统一裁决，不能把冲突处理留给前端

### 3.1 比例

#### 前端表达

建议将当前 `sizeTemplate` 升级为对用户可理解的 `aspectRatio`：

- `9:16 竖版`
- `16:9 横版`
- `1:1 正方形`
- `3:4 打印`

前端仍可保留卡片式选择，但展示“比例 + 用途”。

#### 后端 / 模型映射

服务端维护统一映射表：

| 前端值 | 内部值 | 当前模型请求 | 额外 prompt 影响 |
|---|---|---|---|
| `9:16` | `portrait_9_16` | `sizeTemplate=mobile` | 强化纵向主体、上下留白 |
| `16:9` | `landscape_16_9` | `sizeTemplate=wechat_cover` | 强化横向展开、左右叙事 |
| `1:1` | `square_1_1` | `sizeTemplate=weibo` | 强化中心构图 |
| `3:4` | `print_3_4` | `sizeTemplate=a4` | 强化版式稳定和可打印感 |

建议保留现有 `sizeTemplate` 作为 provider adapter 内部字段，不再直接作为前端主参数。

#### 冲突规则

- 用户显式选了比例时，优先级最高
- 若用户未选比例且 `auto=true`，可根据参考图方向自动选择
- 参考图为横图但用户强选竖图时，不报错，只在 prompt 中强调“竖版重构构图”

#### 默认值

- 默认 `9:16`
- 原因：当前产品主场景明显偏海报流和移动端分享，竖版更稳定

### 3.2 清晰度

#### 前端表达

不要使用假的“超清”开关。建议使用二档或三档：

- `标准`
- `高清`
- `超清`：仅在已接入超分链路时展示

若当前阶段没有超分服务，MVP 只展示 `标准 / 高清`。

#### 后端 / 模型映射

`clarity` 不是当前 Doubao 的原生参数，必须做成复合映射：

| 前端值 | 参数类型 | 服务端真实动作 |
|---|---|---|
| `standard` | 复合策略 | 使用比例默认尺寸，基础质量词，基础负向词 |
| `hd` | 复合策略 | 尽量使用该比例下的较高输出尺寸，增加细节/边缘/材质质量词，增强“模糊、压缩痕迹、锯齿”等负向词 |
| `ultra` | 复合策略 | `hd` 生成后再走超分或放大服务；若未接入超分则降级为 `hd` 并返回 `effectiveClarity=hd` |

MVP 的真实落点应是：

- 调整输出尺寸策略
- 调整 prompt 中的质量增强词
- 调整 negative prompt 的质量约束

而不是前端点一下“高清”但请求体没有任何变化。

#### 冲突规则

- `ultra` 与“快速生成”模式冲突时，以速度优先降级到 `hd`
- `hd/ultra` 会增加耗时和成本，应由服务端返回 `effectiveProfile`

#### 默认值

- 默认 `hd`
- 原因：当前项目目标是“质量提升”，不是极致低成本试错；`hd` 更符合品牌海报场景

### 3.3 自动

#### 前端表达

建议将当前快捷动作里的“自动”收敛为一个明确的开关或模式：

- `自动优化：开 / 关`
- 或 `模式：手动 / 自动 / Agent`

`自动` 不应只是一个文案标签，必须映射为策略执行开关。

#### 后端 / 模型映射

`auto` 不是 provider 参数，而是“决策策略参数”：

- 自动选择风格模板
- 自动补全构图/镜头/光线/材质
- 自动选择比例（仅当用户未显式指定）
- 自动选择清晰度默认档位（例如品牌海报默认 `hd`）
- 自动从参考图吸收配色与氛围

可执行规则示例：

1. 用户未选比例，且参考图方向明确，则跟随参考图方向
2. 用户输入过短，自动补全视觉槽位
3. `brand/product` 类任务自动加入“留白、版式稳定、主体突出”
4. `agent` 模式允许更强推断；`manual` 模式只做轻补全

#### 冲突规则

- 显式参数永远覆盖自动推断
- `manual` 模式下，自动只做安全和最小补全
- `agent` 模式下，自动可以补全更多场景与风格，但仍不可改动主体

#### 默认值

- 新用户默认 `auto=on`
- 专业用户切到 `manual`
- 原因：大多数用户无法自己写出完整视觉 prompt

## 4. 推荐实现结构

### 4.1 建议新增模块 / 文件

建议沿用现有工程结构，在服务端新增以下模块：

- `utils/prompt-engine/schema.js`
  - 定义 `GenerationIntent`、`IntentSummary`、`PromptSpec`
- `utils/prompt-engine/normalize.js`
  - 输入清洗、空值归一、基础词法归一
- `utils/prompt-engine/extract.js`
  - 主体/场景/风格/用途提取
- `utils/prompt-engine/enrich.js`
  - 风格、构图、镜头、光线、材质补全
- `utils/prompt-engine/safety.js`
  - 复用并包装 `sensitive-filter.js`
- `utils/prompt-engine/serialize.js`
  - 结构化结果转最终 prompt/negative prompt
- `utils/parameter-mapping.js`
  - 比例、清晰度、自动模式的统一映射
- `utils/provider-adapters/doubao-image.js`
  - 将内部参数转换成 Doubao 请求体

对现有文件的建议：

- `utils/prompt-builder.js` 逐步降级为兼容入口，内部转调 prompt engine
- `utils/doubao.js` 聚焦 provider adapter，不再承载业务参数解释
- `api/generate.js` 只负责请求校验、调用编排、返回结果

前端建议新增：

- `src/constants/generation-options.js`
  - 比例、清晰度、模式选项常量
- `src/utils/build-generate-payload.js`
  - 只负责 UI 状态到 API payload 的转换

### 4.2 前端与服务层分工

前端负责：

- 收集用户输入和显式参数
- 提供参数选择与解释文案
- 展示优化摘要或实际生效参数
- 做基础校验，不做最终参数裁决

服务层负责：

- Prompt 优化
- 安全过滤
- 参数冲突解决
- provider 参数映射
- 返回 `effectivePromptSpec`、`effectiveProfile`

这样做的原因：

- 防止前端和后端各自拼 prompt，逻辑分叉
- provider 细节不暴露到浏览器
- 后续视频和 Agent 复用同一中间层

### 4.3 API 合约建议

建议 `POST /api/generate` 扩展为：

```json
{
  "rawPrompt": "运动品牌春季海报，女生晨跑，干净有力量",
  "mode": "auto",
  "posterType": "brand",
  "aspectRatio": "9:16",
  "clarity": "hd",
  "auto": "true",
  "style": "cinematic-realism",
  "referenceImageUrl": "https://..."
}
```

服务端返回建议包含：

```json
{
  "imageUrl": "data:image/png;base64,...",
  "promptSpec": {},
  "effectiveProfile": {
    "aspectRatio": "9:16",
    "clarity": "hd",
    "sizeTemplate": "mobile",
    "size": "1024x1792"
  }
}
```

这样前端能展示“实际生效参数”，避免用户误以为所有选择都原样命中模型。

### 4.4 为视频 / Agent 模式预留扩展

关键是复用同一个中间对象 `PromptSpec`。

扩展方式：

- 图片模式：`PromptSpec -> image serializer`
- 视频模式：`PromptSpec + motionSpec -> video serializer`
- Agent 模式：`IntentSummary -> 多步规划 -> PromptSpec`

后续只需增加：

- `motion`：镜头移动、节奏、时长
- `sequence`：分镜或场景段落
- `taskPlan`：Agent 自动拆解结果

而不需要重写整套参数层。

## 5. 落地建议

### 5.1 立即可做的两步

1. 先建立 `parameter-mapping + prompt-engine schema`，把“比例/清晰度/自动”从 UI 文案变成服务端真实参数
2. 再替换 `utils/prompt-builder.js`，输出结构化 `PromptSpec` 和最终 `finalPrompt`

### 5.2 MVP 验收点

- 相同原始输入下，`auto` 生成结果比当前直接透传更稳定
- 比例、清晰度、自动三项都能在响应里看到真实生效结果
- Prompt 优化不会改掉主体，只补全视觉槽位
- 安全过滤仍有效，且不会因补全引入额外风险词

## 6. 核心结论

1. 当前项目最缺的不是更多 UI，而是 `Intent -> PromptSpec -> ProviderRequest` 的中间层
2. Prompt 优化引擎 MVP 应以规则驱动为主，先解决稳定性和可控性，再考虑引入额外 LLM 改写
3. “清晰度”和“自动”都不是当前模型原生参数，必须设计成服务端可执行的复合映射
4. 参数冲突必须由服务端统一裁决，显式参数永远覆盖自动推断
5. 只要先把结构化 prompt 和真实参数层搭好，后续扩展到视频/Agent 会明显更顺
